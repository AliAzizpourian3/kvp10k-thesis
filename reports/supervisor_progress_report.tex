\section{Thesis Recap}
% ============================================================

This thesis investigates \textbf{key-value pair (KVP) extraction} from semi-structured documents (invoices, receipts, forms) using IBM's \textbf{KVP10k} benchmark dataset (ICDAR 2024).
The KVP10k paper argued that encoder-based models such as LayoutLMv3 have limitations for direct KVP linking, and therefore evaluated only a generative LMDX-style baseline rather than empirically benchmarking LayoutLMv3 on KVP10k.

This thesis revisits that claim by combining \textbf{layout-aware modelling} with \textbf{data-centric analysis}.
The research goals from the proposal are:

\begin{enumerate}[nosep]
  \item \textbf{Re-evaluate Layout-Aware Modelling:} Reassess LayoutLMv3 for KVP extraction and extend it with a lightweight Linker module for explicit key--value relationships.
  \item \textbf{Analyze and Categorize Document Layouts:} Apply unsupervised clustering to group documents by structural similarity and explore model performance across layout types.
  \item \textbf{Assess Robustness and Generalization:} Investigate model behaviour under layout distortions (masking, rotation, partial removal) and compare OCR-based (LayoutLMv3) with OCR-free approaches (Donut, PaLI-X, Pix2Struct) in zero-shot and few-shot settings.
  \item \textbf{Derive Analytical Insights:} Examine learning trends, ablation effects, and error patterns. Use embedding visualization (t-SNE, PCA) to explore how key, value, and other text elements are represented inside the model.
\end{enumerate}

The project is organized into five stages (Figure~\ref{fig:pipeline}).
\textbf{Stages~0--3 are complete}; Stage~4 is the next focus.

\section{Completed Work (Stages 0--3)}
% ============================================================

\subsection{Stage 0: Evaluation Protocol}
I adopted the evaluation protocol from the KVP10k paper (Perot et al., 2023):
\begin{itemize}[nosep]
  \item \textbf{Text-only:} Normalized Edit Distance (NED) $< 0.2$ between predicted and ground-truth text.
  \item \textbf{Text + Location:} NED $< 0.2$ \emph{and} bounding-box IoU $> 0.3$.
\end{itemize}

\subsection{Stage 1: Data Ingestion \& Preparation}

The raw KVP10k dataset was not directly model-ready.
Each row contains annotations, a PDF URL, a page number, and a page identifier (\texttt{hash\_name}), but not a complete training sample.
Substantial preparation was required:

\begin{enumerate}[nosep, leftmargin=1.5em]
  \item \textbf{Deduplication:} The same page appears $\sim$5 times in the raw data (one per annotator).
        I grouped by \texttt{hash\_name} and selected the richest usable annotation set, reducing 48{,}280 train rows to 9{,}656 unique pages and 5{,}255 test rows to 1{,}051 unique pages.
  \item \textbf{PDF download \& validation:} For each unique page, I downloaded the source PDF and verified it was a valid PDF (many URLs returned broken files, permission-denied pages, or HTML error pages).
  \item \textbf{Text extraction:} I used PyMuPDF to extract words and bounding boxes from each page's text layer.
        Pages without a text layer (scanned images with no OCR) were discarded---this is the main source of data loss.
  \item \textbf{Annotation fusion:} I matched extracted words against annotation bounding boxes (by spatial overlap) to assign entity labels and reconstruct key--value link supervision.
  \item \textbf{LMDX formatting:} Each word was encoded as \texttt{word x1|y1|x2|y2} in normalized $[0,100]$ coordinates, following the format from the KVP10k paper.
\end{enumerate}

\section{Technical Challenges Encountered}
% ============================================================

\begin{enumerate}[leftmargin=1.5em]

\item \textbf{HPC Offline Compute Nodes.}
  The FAU HPC cluster compute nodes have \emph{no internet access}.
  All Hugging Face models, tokenizers, and processors had to be pre-downloaded on login nodes and cached locally.
  Every \texttt{from\_pretrained()} call required \texttt{local\_files\_only=True}.
  Multiple early SLURM jobs failed with network timeouts before this was identified.

\item \textbf{CUDA / PyTorch / Quantization Compatibility.}
  The cluster runs CUDA~13.0 with A100-40GB GPUs.
  Building a working environment required resolving version conflicts between PyTorch, Transformers, and bitsandbytes (required for 4-bit QLoRA).
  The quantization library needed specific CUDA toolkit alignment that was not straightforward on the shared HPC system.

\item \textbf{Data Preparation Pipeline.}
  As described in Stage~1, the raw KVP10k data required substantial engineering to convert into training-ready format.
  Key difficulties included: broken/inaccessible source PDFs ($\sim$45\% of pages), conflicting duplicate annotations from multiple annotators, and reproducing IBM's preparation workflow with different tooling (PyMuPDF instead of their OCR stack).

\item \textbf{Generative Model Output Parsing.}
  Mistral-7B generates free-form text, and parsing structured KVP output from its responses required robust post-processing.
  The model occasionally produces malformed outputs (incomplete JSON, hallucinated fields, or bounding-box coordinates outside the valid range), all of which had to be handled gracefully during evaluation.

\end{enumerate}

\section{Literature Review Findings}
% ============================================================

In preparation for Stage~4, I conducted a literature review of existing approaches to key information extraction and entity linking in documents:

\begin{itemize}[leftmargin=1.5em]
  \item \textbf{SPADE} (Hwang et al., ACL Findings 2021) formulates information extraction as \emph{spatial dependency parsing}, modelling relationships among text tokens as a graph rather than a sequence tagging problem.
        This is directly relevant to the linking subtask.

  \item \textbf{BROS} (Hong et al., AAAI 2022) uses a BERT-based encoder with relative 2D position encoding and a \emph{subsequent token prediction} head for linking---avoiding pairwise scoring entirely.
        It achieves strong results on FUNSD, SROIE, and CORD without visual features.

  \item \textbf{LMDX} (Perot et al., 2023) uses generative LLMs (PaLM~2-S, Gemini Pro) and is the approach evaluated in the KVP10k paper.
        Their link F1 of 0.659 sets the current benchmark, but relies on a proprietary generative model whose parameter count is undisclosed.

  \item \textbf{LayoutLMv3} (Huang et al., ACM MM 2022) pre-trains on unified text and image masking.
        It achieves entity F1 of 0.91 on FUNSD (form understanding), but the official codebase only provides entity classification---\emph{no relation extraction head is included}.
        Adding a linking module on top of LayoutLMv3 for KVP extraction is an open contribution.
\end{itemize}

These works inform my Stage~4 design: I plan to combine LayoutLMv3's strong document encoding with a linking head inspired by the approaches in SPADE and BROS.

\begin{enumerate}[nosep]
  \item \textbf{Entity Classifier:} Token-level classification into Key / Value / Other.
  \item \textbf{Linking Head:} Learns to pair detected key spans with their corresponding value spans, inspired by biaffine scoring (SPADE) or next-token prediction (BROS).
\end{enumerate}

\subsection{Anticipated Challenges}

Based on the literature review, I expect several challenges in this stage:

\begin{itemize}[nosep]
  \item \textbf{Class imbalance in linking:} The key--value link matrix is extremely sparse (most token/span pairs are not linked). SPADE and BROS both address this with specialized architectures; I may need similar strategies.
  \item \textbf{Scale gap vs.\ generative models:} LayoutLMv3-base has 125M parameters, compared to the KVP10k paper's proprietary generative model (parameter count undisclosed, but substantially larger). The discriminative model may not match the generative F1, which itself would be an interesting finding.
  \item \textbf{Train/test distribution mismatch:} If entity predictions are imperfect at test time, the linker must work with noisy inputs---a known difficulty in pipeline architectures.
\end{itemize}

\subsection{Additional Analyses (from Proposal)}

Beyond the core LayoutLMv3 + Linker experiment, the following analyses are planned as part of Stage~4 and the thesis discussion:

\begin{itemize}[nosep]
  \item \textbf{Robustness testing:} Evaluate model stability under layout distortions (masking, rotation, partial removal).
  \item \textbf{OCR-based vs.\ OCR-free comparison:} Compare LayoutLMv3 with OCR-free approaches (Donut, PaLI-X, Pix2Struct) in zero-shot and few-shot settings, if time permits.
  \item \textbf{Embedding visualization:} Use t-SNE or PCA to visualize how key, value, and other text elements are represented in the LayoutLMv3 hidden space, and how spatial relationships emerge during fine-tuning.
  \item \textbf{Per-cluster performance analysis:} Break down Stage~4 results by the layout clusters identified in Stage~2.
\end{itemize}
